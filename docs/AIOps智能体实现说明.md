# AIOps 智能体实现说明

本文记录当前仓库内 AIOps 智能体的落地实现思路、关键模块、运行链路与约束，便于后续继续演进、排障和交接。

相关文档：

- `docs/AIOps实现方案.md`：偏初版方案与模块拆分
- `docs/AIOps-MCP-Skill-双阶段应答设计.md`：偏双阶段应答设计
- 本文：偏当前代码实现、运行细节与工程约束

## 1. 总体目标

当前 AIOps 智能体的目标不是做一个脱离平台的数据问答机器人，而是把平台已有的资源、告警、工单、任务、事件、容器、中间件等能力统一抽象为 LLM 可调用的工具，再由模型结合工具事实给出回答或生成待执行动作。

核心原则：

- 只通过平台内数据和工具回答，不走前端关键词拼装查询
- 优先使用 LLM tool-calling 决策，不再依赖规则匹配路由
- 高风险动作先生成草稿，再确认执行
- 回答必须尽量基于事实，不允许自由编造
- 问答、工具调用、动作执行都要可审计

## 2. 整体架构

当前实现可拆成 6 层：

1. 前端聊天层  
   `frontend/src/components/aiops/AIOpsChatWidget.vue`

2. AIOps API 层  
   `backend/aiops/views.py`

3. 智能体调度层  
   `backend/aiops/services.py`

4. 平台内置 MCP 工具层  
   由 `services.py` 中的工具注册与执行逻辑统一暴露

5. 外部 MCP 接入层  
   通过 `AIOpsMCPServer` 配置接入 HTTP / stdio MCP

6. Skill 与回答整形层  
   约束回答结构、补足可读性与一致性

## 3. 一次问答是怎么跑起来的

以“分析生产 order-center 最近异常”为例，当前链路如下：

1. 前端发送问题到 `send_message` 或 `send_message_async`
2. 后端创建用户消息和占位助手消息
3. 调度层加载：
   - 默认模型配置
   - 启用的 MCP
   - 启用的 Skill
   - 历史消息
4. 第一阶段 LLM 做 tool-calling 规划
5. 后端按工具白名单真正执行 MCP / 平台工具
6. 收集结构化工具结果，形成事实集
7. 生成一版基于事实的回答草稿
8. 第二阶段结合 Skill 做回答整形
9. 若整形结果不合格，再回退到代码兜底模板
10. 返回最终消息、思考过程、工具调用、待确认动作

其中“思考过程”不是模型原始推理全文，而是平台定义的一组可控进度事件，例如：

- 加载 MCP 与 Skill
- 模型规划
- 生成工具计划
- 生成回复
- Skill 模板整形

这样既能提升可观测性，又不会暴露不可控的底层推理细节。

## 4. 为什么现在强调 MCP + Skill

### 4.1 MCP 的职责

MCP 负责把平台能力标准化为模型可调用的工具。

当前平台侧重点是“平台内置 MCP”，本质上是把现有业务能力映射成统一工具接口，而不是让 LLM 直接访问数据库或拼 REST 查询。

当前已覆盖的内置 MCP 方向包括：

- CMDB MCP
- 可观测性 MCP
- 工单系统 MCP
- 任务中心 MCP
- 事件墙 MCP
- 容器管理 MCP
- 中间件 MCP

外部 MCP 也已支持接入，例如：

- `n9e-mcp-server`
- `skywalking-mcp`
- `mcp-grafana`

### 4.2 Skill 的职责

Skill 不负责查数据，主要负责两件事：

- 约束模型该怎么回答
- 让同类问题的输出结构更稳定

例如：

- 告警/异常分析问题：优先输出“结论、依据、建议操作”
- 工单汇总问题：优先输出“范围、数量、明细、建议”
- 任务生成问题：优先输出“目标、动作、风险、确认提示”

## 5. 为什么不是规则匹配

之前系统里曾混入过部分规则化意图兜底，容易出现两个问题：

- 用户自然语言稍微变化，命中就不稳定
- 平台看起来像“会问答”，实际是关键词查表

当前实现方向已经收敛为：

- 问题理解：交给 LLM
- 工具选择：交给 LLM tool-calling
- 工具执行：由平台后端控制
- 回答整形：由 Skill + LLM 完成

也就是说，代码负责提供工具、权限、安全边界和兜底，不负责替模型做意图分类。

## 6. 平台内置工具是怎么落到代码里的

核心代码位于：

- `backend/aiops/services.py`

这里做了几件关键事情：

### 6.1 工具注册

把平台内方法注册成统一工具集合，例如：

- `query_cmdb_items`
- `query_hosts`
- `query_alerts`
- `query_logs`
- `query_traces`
- `query_recent_changes`
- `query_workorders`
- `query_task_center`
- `query_event_wall`
- `query_container_assets`
- `query_k8s_cluster_summary`
- `query_middleware_assets`
- `generate_host_task`

### 6.2 工具执行

模型只负责给出“想调用哪个工具、参数是什么”，真正执行永远在后端。

好处：

- 可做权限控制
- 可做参数清洗
- 可做审计
- 可做超时和异常控制
- 可统一接内置工具和外部 MCP

### 6.3 查询清洗

模型生成的查询有时会带噪音，例如把“日志、链路、变更”一起塞给单个工具。

因此后端仍会做轻量清洗，但只做参数净化，不做意图改写。  
重点是：

- 保留环境、服务名、系统、IP 等有效过滤条件
- 去掉明显跨工具噪音词
- 不把自然语言改写成规则匹配分支

## 7. 双阶段回答的必要性

单阶段 tool-calling 常见问题：

- 工具调对了，但最终回答太短
- 有事实，但表达不稳定
- 不同问题格式差异太大
- 有时会漏掉“建议操作”“可继续查看”

所以当前实现采用双阶段：

### 第一阶段：取证

- LLM 负责理解问题
- 选择工具
- 收集事实

### 第二阶段：整形

- LLM 根据事实 + Skill 模板重写回答
- 保证回答更像一个稳定的企业平台助手

### 兜底层

如果整形失败，再回退到代码模板，保证至少可用。  
目标不是长期依赖兜底，而是让 Skill 整形成为主路径，代码兜底成为最后保险。

## 8. 生成任务为什么必须二次确认

任务生成与查询类问题不同，属于动作型能力。

当前实现区分：

- 查询类：直接返回结果
- 生成类：先生成任务草稿
- 执行类：用户确认后再真正落任务

相关逻辑集中在：

- `backend/aiops/services.py`

这里会把模型输出的任务草稿转成平台待确认动作，再由前端点确认，最终写入任务中心。

这样可以避免：

- 用户只是咨询，系统却直接创建任务
- 目标主机识别不清，误生成任务
- 高风险指令未经确认直接执行

## 9. 会话、审计与 demo 账号同步

当前 AIOps 还做了几个偏产品化的实现：

### 9.1 会话同步

`admin` 的演示会话会同步镜像给 `demo`，用于演示账号展示。

相关逻辑：

- `backend/aiops/services.py`

包括：

- 同步会话
- 同步消息
- 同步待确认动作
- 同步时间戳

### 9.2 审计

智能体配置里的审计页会展示：

- 最近会话
- 最近工具调用
- 最近动作

删除会话也带权限控制和审计记录。

### 9.3 demo 账号限制

为了演示安全，当前已限制 demo 账号真正发起问答。  
前后端会同时拦截，并提示：

`演示账号问答权限已临时关闭，如需体验请联系作者：592095766@qq.com`

## 10. 前端体验层的关键设计

关键组件：

- `frontend/src/components/aiops/AIOpsChatWidget.vue`

当前前端重点不只是“能聊”，还强调：

- 支持会话历史
- 支持异步轮询
- 支持流式显示
- 支持思考过程折叠
- 支持待确认动作卡片
- 支持引用/工具结果展示

另外顶部导航、右下角浮窗、AIOps 页面入口会共用同一个打开事件：

- `sxdevops-aiops-open`

这样可以保证多个入口打开的是同一个智能助手面板，而不是多套实现。

## 11. 当前已经落地的几个关键修复点

这段时间围绕“让 6 个建议问题真正跑通”做过几类关键修复：

### 11.1 告警查询修复

- 修复严重、未确认等过滤条件
- 补充适合异常分析的告警 demo

### 11.2 工单查询修复

- 修复只查到事务工单、查不到应用发布的问题
- 优化“最近交易系统生产有哪些工单”的聚合结果

### 11.3 K8s 查询修复

- 补充异常 Pod demo
- 优化“app-prod-k8s 集群有没有异常的 pod”问答结果

### 11.4 CMDB 与任务生成修复

- 修复 CMDB 查询 IP / 主机识别不稳定
- 修复 Redis 巡检任务生成时的目标主机识别问题
- 保证任务创建先确认再写入任务中心

### 11.5 回答显示修复

- 修复流式显示时答案被截断、刷新后才完整的问题
- 优化“可继续查看”统一格式
- 优化 Markdown 显示

## 12. 当前已知边界

当前实现仍有几个工程边界需要明确：

- Skill 整形已是主方向，但仍保留代码兜底
- 外部 MCP 目前以“先接入可配置、后逐步真实跑通”为主
- 部分 demo 数据仍承担演示职责，不等同于生产真实数据
- 思考过程是平台事件流，不是原始 Chain-of-Thought
- 智能体能力强依赖默认模型、启用工具和数据质量

## 13. 后续建议演进方向

建议继续沿以下方向推进：

1. 让不同问题类型拥有独立 Skill 模板族  
   例如告警分析、工单汇总、成本分析、任务生成、K8s 异常分析

2. 让回答整形完全模板化配置  
   减少代码里硬编码回答样式

3. 继续扩展平台内置 MCP  
   把更多平台能力都统一抽象为工具

4. 增强外部 MCP 健康检查  
   包括连通性、工具发现、超时与鉴权诊断

5. 增强审计可追踪性  
   支持按会话回放完整“问题 → 工具 → 整形 → 动作”链路

## 14. 一句话总结

当前 AIOps 智能体的实现，本质上是：

**用 LLM 做理解和规划，用 MCP 提供事实能力，用 Skill 约束回答结构，用平台后端负责安全、审计、执行与兜底。**
