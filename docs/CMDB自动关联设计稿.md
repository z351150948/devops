# CMDB自动关联设计稿

## 1. 背景与目标

当前资源地图中的 CI 和关系，主要依赖人工录入、导入脚本或演示数据生成。随着应用、数据库、中间件、第三方服务逐步增多，纯手工维护会带来几个明显问题：

- 新增系统上线后，资源关系不能及时补齐
- 应用迁移、扩容、改配置后，拓扑关系容易过期
- 不同来源的数据口径不一致，导致同一资源重复建模
- 运维、研发、架构视角不同，关系语义容易混淆

本设计稿目标是为 CMDB 增加一套“自动关联”能力，让系统可以基于配置、运行时观测、日志、注册信息等证据，自动识别并推荐 CI 之间的关系，再由人工确认或按规则自动生效。

预期达到的效果：

- 自动补齐应用、主机、数据库、中间件、外部服务之间的关系
- 区分“运行在”“连接到”“依赖”三类核心语义
- 通过规则、评分、证据链提高可解释性和可控性
- 支持先推荐、后确认，再逐步走向高置信自动生效

## 2. 设计范围

本期聚焦资源地图中最常见的 CI 和关系，不追求一次覆盖所有资产类型。

### 2.1 CI 范围

- 应用服务
- 主机 / 虚拟机 / 容器节点
- Kubernetes 集群内工作负载
- MySQL / Redis / MQ 等中间件实例
- Nginx / 网关 / 代理类组件
- 外部服务（如安心签、OCR、企查查、金蝶等）

### 2.2 关系范围

- `runs_on`：运行在
- `connects_to`：连接到
- `depends_on`：依赖

### 2.3 暂不纳入

- 复杂网络拓扑中的 L2 / L3 物理链路
- 完整调用链画像替代 APM 平台
- AI 自动生成全部关系且不做证据约束

## 3. 关系语义定义

自动关联能否好用，关键在于关系语义要稳定，否则同一条边会被不同人理解成不同含义。

### 3.1 `runs_on`：运行在

表示某个逻辑 CI 实际承载于某个计算资源上。

典型示例：

- `trade-service -> host-trade-01-prod`
- `user-center -> k8s-node-01`
- `mall-job -> k8s-cluster-prod`

可接受证据：

- Agent 上报的进程、部署包、服务信息
- Kubernetes 中 Pod / Deployment 到 Node 的映射
- 应用实例和宿主机的注册关系
- systemd / supervisor / 容器运行信息

### 3.2 `connects_to`：连接到

表示一个 CI 在网络或协议层面主动访问另一个 CI。

典型示例：

- `trade-service -> mysql-master-prod`
- `nginx-gateway -> user-center`
- `user-center -> qichacha-openapi`

可接受证据：

- APM / Trace 中的下游访问记录
- 连接采样、eBPF、Service Mesh 观测
- 配置文件中的数据库、Redis、MQ、HTTP endpoint
- 反向代理或网关路由配置

### 3.3 `depends_on`：依赖

表示某个 CI 在业务或能力上依赖另一个 CI，但不一定总能直接看到稳定的连接证据。

典型示例：

- `trade-service -> 安心签`
- `trade-service -> 金蝶云`
- `user-center -> OCR`

可接受证据：

- 业务配置中的服务开关、供应商标识、SDK 配置
- 代码仓库中明确的第三方服务接入声明
- 产品配置、审批流程、集成清单
- 人工确认的长期依赖关系

### 3.4 三类关系的区别

- `runs_on` 关注“承载关系”，回答“这个应用跑在哪”
- `connects_to` 关注“通信关系”，回答“这个系统连到了谁”
- `depends_on` 关注“能力依赖”，回答“这个系统依赖谁提供能力”

同一对 CI 可以同时存在多种关系，但在展示和推理时应分别处理。

## 4. 自动关联总体流程

整体流程建议设计为：

```text
数据采集
  -> 数据清洗
  -> CI 标准化与匹配
  -> 关系候选生成
  -> 证据归并
  -> 置信度评分
  -> 自动生效或待确认
  -> 拓扑展示与持续校正
```

这个流程强调两点：

- 先生成“候选关系”，不要直接写死进正式关系
- 每条关系都要保留“来源 + 证据 + 分数 + 状态”

## 5. 数据来源设计

### 5.1 静态数据源

适合发现结构化、可重复计算的依赖信息。

- 应用配置
  - 数据库连接串
  - Redis / MQ 地址
  - 第三方服务域名、AK/SK 配置项、endpoint
- 注册中心 / 配置中心
  - Nacos / Apollo / Spring Cloud Config
- 发布平台
  - 应用部署目标、环境、命名空间
- Kubernetes
  - Deployment / Pod / Service / Endpoint / Ingress
- 网关配置
  - 路由、上游服务、域名映射

### 5.2 动态数据源

适合发现真实运行中的连接关系。

- APM / Trace
  - 上下游服务调用
  - 接口耗时
  - HTTP / RPC 目标地址
- 主机连接采样
  - netstat / ss / conntrack
  - eBPF 网络观测
- 日志
  - Nginx access log
  - API Gateway 日志
- 数据库 / 中间件审计
  - RDS 审计日志
  - Redis / MQ 客户端来源

### 5.3 管理面数据源

用于提高关系解释性和命名准确性。

- CMDB 现有资产台账
- 应用负责人、系统归属、环境信息
- 第三方服务字典
- 手工确认或驳回记录

## 6. CI 标准化与匹配

自动关联的难点，不只是发现关系，还在于“把证据映射到正确的 CI”。

### 6.1 统一标识策略

建议为不同类型 CI 设计统一匹配键：

- 应用：应用编码 + 环境 + 命名空间
- 主机：IP / 主机名 / 云实例 ID
- 数据库：实例名 + 地址 + 端口
- 中间件：集群名 + 地址 + 端口
- 外部服务：标准服务编码 + 域名 / endpoint 模式

### 6.2 标准化规则

- 域名统一小写，去协议头与尾部斜杠
- IP:Port 统一拆分并规范存储
- 别名映射到标准 CI 名称
- K8s Service / Pod / Deployment 建立映射表
- 同一供应商的多个入口域名归并为一个外部服务 CI

### 6.3 第三方服务字典

外部服务建议维护一张标准字典表，用于把零散配置收敛为可识别 CI。

示例：

| 关键字 / 域名特征 | 标准 CI 编码 | 展示名称 |
| --- | --- | --- |
| `qcc.com` / `企查查` | `qichacha-openapi` | 企查查 |
| `kingdee` / `金蝶` | `kingdee-cloud` | 金蝶云 |
| `ocr` | `ocr-platform` | OCR服务 |
| `anxinqian` / `安心签` | `anxinqian-api` | 安心签 |

## 7. 关系生成规则

### 7.1 `runs_on` 规则

规则示例：

- 若应用实例清单中记录宿主机，则生成 `应用 -> 主机`
- 若 Pod 已绑定 Node，则生成 `工作负载 -> Node`
- 若部署记录包含目标集群 / 命名空间，则生成 `应用 -> 集群`

建议优先级：

1. Agent / 宿主采集
2. Kubernetes 实时信息
3. 发布平台记录
4. 人工维护回填

### 7.2 `connects_to` 规则

规则示例：

- 解析 JDBC URL 命中数据库 CI，生成 `应用 -> MySQL`
- 解析 Redis / MQ 地址命中目标 CI，生成 `应用 -> 中间件`
- APM 中连续出现稳定下游地址，生成 `服务 -> 服务`
- 网关路由指向后端服务，生成 `网关 -> 应用`
- 命中第三方服务字典域名，生成 `应用 -> 外部服务`

建议增加窗口期和阈值，例如：

- 最近 7 天至少出现 20 次连接
- 同一来源应用在 3 个以上时间窗口都出现

### 7.3 `depends_on` 规则

规则示例：

- 配置中存在第三方供应商开关或 SDK 标识，生成依赖候选
- 业务接入清单中声明集成服务，生成依赖候选
- 人工确认 `connects_to` 且长期稳定，可同步推导为 `depends_on`

建议限制：

- `depends_on` 不能只靠一次网络连接生成
- 需要配置、字典、人工确认中的至少一种强证据

## 8. 置信度评分模型

每条候选关系建议都计算置信度，便于自动生效和人工审核。

### 8.1 评分维度

- 证据来源权重
  - Kubernetes / Agent / 发布平台：高
  - 配置解析：中高
  - APM / eBPF：中高
  - 日志关键字：中
  - 文本匹配：低
- 证据数量
  - 多来源交叉验证得分更高
- 时间连续性
  - 多个时间窗口持续出现得分更高
- 匹配准确性
  - 精确命中字典或实例标识得分更高
- 人工反馈
  - 被确认加分，被驳回降权

### 8.2 评分示例

- 90 分以上：可自动生效
- 70-89 分：待人工确认
- 50-69 分：仅作推荐，不入正式关系
- 50 分以下：丢弃或继续观察

### 8.3 证据展示

每条候选关系应支持查看证据详情，例如：

- 命中的配置项
- 最近出现时间
- 采集来源
- 命中次数
- 字典映射结果

## 9. 生命周期管理

关系不是一次生成后永久不变，需要具备生命周期管理能力。

### 9.1 状态设计

- `pending`：待确认
- `active`：已生效
- `rejected`：已驳回
- `stale`：过期待清理

### 9.2 失效机制

- 若关系在设定窗口内持续无证据，则降为 `stale`
- 若被人工驳回，则在一段时间内禁止同规则重复推荐
- 若 CI 被下线、合并或归档，则自动处理关联关系

### 9.3 历史留痕

建议保留：

- 首次发现时间
- 最近确认时间
- 最近命中时间
- 最近驳回原因
- 自动 / 人工操作日志

## 10. 人工确认与运营流程

自动关联不应一开始就完全无人值守，建议先建立“推荐 - 确认 - 生效”的运营闭环。

### 10.1 待确认工作台

可提供以下能力：

- 按环境、业务线、CI 类型筛选
- 查看候选关系、置信度、证据摘要
- 一键确认、驳回、忽略
- 批量确认同类关系

### 10.2 驳回反馈

驳回时可记录原因：

- 误匹配
- 已下线
- 临时连接，不构成依赖
- 应归属为其他 CI

这些反馈可用于后续规则调优。

## 11. 数据模型建议

建议在现有 CMDB 关系基础上，补充一层“候选关系”和“证据表”。

### 11.1 候选关系表

核心字段建议：

- 源 CI ID
- 目标 CI ID
- 关系类型
- 关系状态
- 置信度
- 来源规则
- 首次发现时间
- 最近发现时间
- 是否自动生效

### 11.2 关系证据表

核心字段建议：

- 候选关系 ID
- 证据来源类型
- 原始值
- 标准化值
- 命中时间
- 权重
- 详情 JSON

### 11.3 字典与映射表

- 第三方服务字典表
- 域名 / endpoint 映射表
- 应用别名映射表
- 驳回黑名单规则表

## 12. 页面与交互建议

### 12.1 资源地图展示

在现有拓扑中，可增加以下能力：

- 连线区分来源：人工 / 自动 / 待确认
- 边上显示关系类型和置信度
- 支持仅查看自动发现关系
- 支持筛选“包含外部服务”的依赖链

### 12.2 详情侧栏

点击关系后显示：

- 关系类型
- 当前状态
- 置信度
- 证据列表
- 最近命中时间
- 确认 / 驳回入口

### 12.3 运维视图

可增加几个常用看板：

- 新发现未确认关系数
- 最近 7 天新增外部服务依赖
- 高风险第三方依赖分布
- 证据长期缺失的陈旧关系

## 13. 分阶段实施建议

### 第一阶段：低成本落地

- 建第三方服务字典
- 解析应用配置中的数据库、Redis、MQ、HTTP endpoint
- 生成 `connects_to` 和部分 `depends_on` 候选
- 增加候选关系表和待确认页面

### 第二阶段：补齐运行关系

- 接入发布平台、Agent、Kubernetes 数据
- 自动补齐 `runs_on`
- 把应用、主机、集群的承载链打通

### 第三阶段：增强运行时可信度

- 接入 APM / Trace
- 接入连接采样或 eBPF 数据
- 引入多证据评分模型

### 第四阶段：自动治理

- 对高置信关系自动生效
- 对低置信关系自动淘汰
- 建立驳回反馈闭环与规则自学习机制

## 14. 风险与控制措施

### 14.1 误关联风险

原因：

- 域名命中歧义
- 临时连接被误识别为长期依赖
- 一套配置被多个环境复用

控制措施：

- 强制标准化与环境隔离
- 引入多证据交叉校验
- 先推荐、后确认

### 14.2 数据缺失风险

原因：

- 未接入配置中心或 APM
- 日志采样不足
- 第三方服务配置不规范

控制措施：

- 先从高价值系统试点
- 建立配置规范与字典维护机制
- 支持人工补录和回填

### 14.3 运维成本风险

原因：

- 候选关系过多，人工确认压力大

控制措施：

- 先限制在核心环境、核心应用
- 建立批量确认与规则模板
- 对低分候选直接过滤

## 15. 成功指标

建议用以下指标衡量自动关联效果：

- 核心应用拓扑覆盖率
- 自动生成关系占比
- 人工确认通过率
- 误报率 / 驳回率
- 陈旧关系清理率
- 外部服务依赖识别准确率

## 16. 推荐落地顺序

如果下次要开始实现，建议按下面顺序推进：

1. 先补“第三方服务字典 + 关系证据表”
2. 再做“配置扫描 -> CI 匹配 -> 候选关系生成”
3. 再补“待确认页面和人工反馈流程”
4. 再把“关系来源、置信度、状态”接入资源地图
5. 最后接入 Trace / 连接采样这类动态数据源

这样可以先用最小成本跑通闭环，再逐步提高准确率和自动化程度。
