# 多云资源拓扑现状与第三期待办

## 1. 文档目的

本文档说明当前多云资源拓扑功能的真实状态，区分：

- 目前哪些节点和关系来自真实云环境
- 目前哪些连线仍然是规则推导或演示数据
- 第三期如果要做成“真实云资源拓扑”，需要补哪些能力

适用范围：

- 后端拓扑生成逻辑：`backend/multicloud/services.py`
- 云厂商 SDK 接入层：`backend/multicloud/sdk_adapters.py`
- 前端拓扑展示页：`frontend/src/views/MultiCloudManage.vue`


## 2. 当前实现总览

当前拓扑页面调用接口：

- `GET /api/multicloud/topology/`

接口由后端 `build_topology()` 生成图结构，前端只负责布局和渲染。

当前图结构分为四层：

1. 云账号
2. 环境
3. 资源分层桶节点
   - 计算层 `compute`
   - 数据层 `data`
   - 网络层 `network`
4. 具体资源节点

其中第 3 层“资源分层桶节点”是为了可视化而加的中间层，并不是云厂商原生资源对象。


## 3. 当前节点来源

### 3.1 云账号节点

来源：系统内 `CloudCredential`

说明：

- 这是平台内部真实配置数据
- 无论是 demo 模式还是真实 SDK 模式，这一层都是真实存在的管理对象


### 3.2 环境节点

来源：系统内 `CloudEnvironment`

说明：

- 这是平台内部定义的环境对象
- 环境和云账号之间的关系是系统配置关系，不依赖云厂商 API 自动发现


### 3.3 具体资源节点

来源：系统内 `CloudAsset`

`CloudAsset` 的数据来源有两种：

1. Demo 模式
   - 后端按内置模板生成模拟资源
2. 真实 SDK 模式
   - 后端调用云厂商 SDK 拉取资源清单，再同步到 `CloudAsset`

因此：

- 在 demo 模式下，资源节点是演示数据
- 在真实 SDK 模式下，资源节点可以是真实云资源


## 4. 当前连线是怎么画出来的

### 4.1 云账号 -> 环境

这是系统真实关系。

来源：

- `CloudCredential` 和 `CloudEnvironment` 的归属关系

特点：

- 这条线不是推导的
- 只要环境归属某个账号，这条线就会存在


### 4.2 环境 -> 计算/数据/网络

这是可视化分层关系，不是云厂商真实拓扑关系。

来源：

- 后端在拓扑生成时，按资源类型把资源归入三个桶节点

作用：

- 让画面更稳定
- 让图更像架构图

限制：

- 这层节点不是真实云对象
- 这条线不代表云上真实依赖


### 4.3 分层桶 -> 具体资源

这是基于真实或 demo 资源清单生成的“分组展示关系”。

说明：

- 如果 `CloudAsset` 来自真实 SDK 同步，那么资源节点本身是真实的
- 但这条边仍然只是“该资源属于某一层”的展示边
- 它不代表资源之间的真实网络/依赖关系


### 4.4 资源 -> 资源

这是当前最重要的限制。

目前资源之间的边不是从云厂商 API 直接拿到的真实依赖关系，而是后端静态规则推导。

当前规则位于：

- `backend/multicloud/services.py` 中的 `RESOURCE_LINKS`

现有规则示例：

- `slb -> ecs / k8s`
- `ecs -> rds / redis`
- `k8s -> rds / redis / oss`
- `nat -> ecs / k8s`
- `security_group -> ecs / rds / redis`

这意味着：

- 当前资源间连线是“启发式关系”
- 不是云厂商返回的真实拓扑
- 更适合 demo 展示和概览，不适合作为精确依赖依据

### 5.2 华为云

- ECS 实例清单同步

- 真实模式下可以得到真实 ECS 资源节点
- 但尚未完整拉取 ELB / RDS / Redis / VPC / 子网 / 安全组 / EIP 等资源

## 6. 当前模式下“真实度”怎么理解

### 6.1 Demo 模式

真实度最低。

特点：

- 账号和环境是平台真实对象
- 资源是演示数据
- 资源关系是规则推导


### 6.2 真实 SDK 模式

真实度中等。

特点：

- 账号和环境是真实对象
- 已同步到库的资源节点可以是真实云资源
- 资源之间的关系大部分仍然不是云上原生依赖，而是规则推导


### 6.3 真正的“真实拓扑”

要达到这个标准，需要做到：

- 真实资源清单同步
- 真实资源关系采集
- 用真实关系建边，而不是靠类型规则猜边


## 7. 当前缺口

第三期前，主要缺口有四类。

### 7.1 资源类型覆盖不足

当前真实模式主要稳定覆盖 ECS。

还缺少重点资源类型：

- VPC
- 子网 / 交换机
- 安全组
- SLB / ELB
- RDS
- Redis
- EIP
- NAT
- OSS / OBS 等对象存储


### 7.2 缺少真实关系字段

即使未来把资源都同步进来，如果没有关系字段，也无法画真实拓扑。

需要采集的典型关系字段包括：

- ECS 所属 `vpc_id`
- ECS 所属 `vswitch_id` / `subnet_id`
- ECS 绑定的 `security_group_ids`
- EIP 绑定目标
- NAT 关联 VPC / 子网 / EIP
- SLB 后端实例列表
- RDS 所属网络信息
- Redis 所属网络信息


### 7.3 缺少关系模型

当前系统只有 `CloudAsset`，没有专门的“资源关系”模型。

这会导致：

- 关系难持久化
- 拓扑边只能现场推导
- 不利于审计、比对、漂移检测和 CMDB 对齐


### 7.4 缺少关系可信度分层

后续真实拓扑里，边应该区分来源，例如：

- `observed`：云厂商 API 直接返回
- `derived`：根据真实字段推导
- `inferred`：启发式猜测

当前没有这层标识。


## 8. 第三期待办

建议分三步落地。

### 第一步：补齐真实资源清单

优先云厂商：

1. 阿里云
2. 华为云

优先资源类型：

1. VPC / 交换机 / 安全组
2. ECS
3. SLB
4. RDS / Redis
5. EIP / NAT

目标：

- 不只是同步主机
- 先把“网络骨架”和“核心中间件”同步进来


### 第二步：采集真实关系字段

在 SDK adapter 同步时，把关系信息写进 `CloudAsset.metadata`，例如：

- `vpc_id`
- `subnet_id` / `vswitch_id`
- `security_group_ids`
- `backend_instance_ids`
- `eip_bind_target`

目标：

- 为真实拓扑建边准备原始数据


### 第三步：引入关系模型

建议新增模型，例如：

- `CloudAssetRelation`

建议字段：

- `source_asset`
- `target_asset`
- `relation_type`
- `source_provider`
- `source_environment`
- `evidence`
- `confidence`
- `discovered_at`

关系类型示例：

- `located_in_vpc`
- `located_in_subnet`
- `protected_by_security_group`
- `attached_eip`
- `backend_of_slb`
- `depends_on_rds`
- `depends_on_redis`

目标：

- 拓扑接口直接读关系表画边
- 不再依赖静态 `RESOURCE_LINKS`


## 9. 第三期推荐落地顺序

建议优先从阿里云开始做一条完整链路：

### 9.1 阿里云优先链路

1. VPC
2. VSwitch
3. Security Group
4. ECS
5. SLB
6. RDS / Redis

先完成以下真实边：

- VPC -> VSwitch
- VSwitch -> ECS
- SecurityGroup -> ECS
- SLB -> ECS
- ECS -> RDS
- ECS -> Redis


### 9.2 华为云第二条链路

对应资源：

- VPC
- Subnet
- Security Group
- ECS
- ELB
- RDS / DCS


## 10. 第三期完成后的目标状态

完成后，拓扑图应具备以下能力：

- 节点大部分来自真实云资源同步
- 核心边来自真实 API 或真实字段推导
- 每条边能标明来源和可信度
- 可以支持环境间对比、漂移检测、CMDB 关联
- 可以从“展示型拓扑”升级到“治理型拓扑”


## 11. 当前结论

当前版本适合：

- 多云资源总览
- 账号/环境/资源分层展示
- demo 演示
- 初步真实资产接入

当前版本不应被视为：

- 云厂商原生真实依赖拓扑
- 严格准确的网络关系图
- 可以直接用于根因分析的依赖拓扑

一句话总结：

- 当前资源节点可以逐步真实化
- 当前资源连线仍以规则推导为主
- 第三期的核心工作不是“继续美化图”，而是“建立真实关系数据模型并按真实关系建边”
