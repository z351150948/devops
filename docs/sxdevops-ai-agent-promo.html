Why Now

传统运维的核心问题：信息和动作被拆散了

告警散

告警中心看到红点，但缺少日志、Trace、最近变更和资产上下文。

事件散

发布、执行、失败、审批、关键写操作散落在不同页面，复盘成本高。

任务散

巡检、命令、Playbook 与主机权限脱节，动作入口不统一。

经验散

排障路径依赖个人记忆，结论难沉淀，下一次仍要重新查。

SxDevOps AI Agent 的定位不是再做一个聊天框，而是把平台已有能力包装成可控、可审计、可确认的智能工作流。

Product Map

SxDevOps AI Agent = 可观测性 + 事件中心 + 任务中心 + AIOps

01 可观测性聚合告警、日志、链路追踪、Grafana 大屏，形成一条连续排障路径。
02 事件中心聚焦最终执行结果与关键写操作，按系统、环境、应用快速收敛故障范围。
03 任务中心查询最近任务，生成主机巡检、服务检查、批量命令任务草稿，确认后执行。
04 AIOpsLLM 负责理解与规划，MCP 提供事实，Skill 约束输出，后端负责安全与审计。

四个能力不是并排功能，而是一条链路：可观测性取证，事件中心复盘，任务中心行动，AIOps 把它们组织成可解释的处理过程。

Two-stage Answer

一次问答链路：先取证，再整形，最后可审计

让最终回答稳定、可读、可追溯，不因为模型自由发挥丢失工具事实。

01

用户提问

例如：分析生产 order-center 最近异常。

02

LLM 规划

根据工具 schema 选择告警、日志、Trace、事件中心、任务工具。

03

平台执行

后端按白名单执行工具，做权限、参数清洗、超时控制。

04

事实汇总

结构化工具结果形成证据集和代码兜底草稿。

05

Skill 整形

按模板输出结论、依据、建议操作和可继续查看项。

Before

人工排障路径

  • 1打开告警平台，看 PromQL 和标签。
  • 2去日志中心查错误和关键字。
  • 3到链路追踪定位慢调用和异常 Span。
  • 4翻最近发布、任务和事件，再手工写结论。
After

AI Agent 处理路径

  • ✓自然语言发起分析，自动调用告警、日志、Trace、事件中心。
  • ✓输出结论、依据、风险和建议操作，证据可回看。
  • ✓需要巡检时生成任务草稿，等待用户确认。
  • ✓确认后进入任务中心，执行结果继续回写事件中心。

Safety

安全护栏：让智能体可用，也可控

RBAC 权限

后端接口、前端路由、菜单和 WebSocket 场景统一做权限收敛。

工具白名单

模型只能调用平台注册过的工具，真正执行由后端完成。

动作确认

任务生成先进入待确认动作，用户确认后才写入任务中心。

审计回放

会话、工具调用、待确认动作和关键事件都可追踪。

原则：回答基于事实，不凭空编造；动作经过确认，不越权执行；过程留下记录，不做黑盒运维。

模板族

处置编排

在只读诊断后，接入审批、命令模板、Runbook 和任务编排，形成低风险自动化闭环。

  • 1先建立可信事实链路：资产准、告警准、事件准、任务准。
  • 2再让 AI 负责证据整合、结构化输出和动作草稿生成。
  • 3最后把高频、低风险动作纳入自动化闭环。
