# 二次开发工作流

本文件描述本仓库的二次开发分支策略、变更准入与合并流程。所有功能改动、Bug 修复、配置调整、文档修订都应按本工作流进入版本库。

## 1. 分支定义

| 分支 | 角色 | 保护级别 | 内容来源 |
| --- | --- | --- | --- |
| `main` | 唯一受保护的稳定分支 | 受保护，仅接受来自 `test` 的合并 | 上游 fork 默认分支，已完成全部回归与验证 |
| `test` | 集成测试与日常开发主线 | 推荐受保护，不直接接受外部 push | 二次开发的常规工作分支 |
| `feature/*` | 个人特性分支（本地） | 不保护 | 从 `test` 切出，本地提交，不直接推送到 `test` |

> 默认基线：`main`。所有变更的起点都是 `main` 或 `test`，不得在 `main` 上直接修改。

## 2. 分支生命周期

### 2.1 首次进入

```bash
git clone https://github.com/z351150948/devops.git
cd devops
git switch -c test origin/test   # 拉取并切到 test 分支
git push -u origin test          # 已存在则跳过
```

### 2.2 日常开发

```bash
git switch test                  # 切回开发主线
git pull --rebase origin test    # 同步远端

git switch -c feature/<topic>    # 从 test 切出特性分支
# 编码、调试、自测
git add <paths>
git commit -m "<type>(<scope>): <subject>"
git push -u origin feature/<topic>
```

### 2.3 进入 test

```bash
# 在 GitHub 上从 feature/<topic> 向 test 发起 PR
# PR 通过 CI 与代码审查 → 合入 test
git switch test
git pull --rebase origin test
git branch -d feature/<topic>     # 本地清理
git push origin --delete feature/<topic>   # 远端清理
```

### 2.4 由 test 升级到 main

只有当 `test` 满足以下全部条件时，才允许升级到 `main`：

1. [ ] 后端测试全部通过：`cd backend && python manage.py test`
2. [ ] 前端构建通过：`cd frontend && npm run build`
3. [ ] `docker compose up -d --build` 起服一次冒烟，主页面可登录、可观测页可打开
4. [ ] 涉及 RBAC、菜单、路由、WebSocket 的改动已在后端 `rbac/registry.py` 注册新权限码
5. [ ] 中文 UI / 接口文案已做乱码自检（UTF-8）
6. [ ] changelog 已更新（见第 4 节）
7. [ ] 由维护者在 GitHub 上发起 `test → main` 的 PR 并完成评审

合并命令（仅维护者在本地使用，普通协作者无需执行）：

```bash
git switch main
git pull --rebase origin main
git merge --no-ff test -m "release: merge test into main"
git push origin main
git push origin --delete <已合并的临时分支>   # 可选
```

## 3. 提交流程规约

### 3.1 提交信息

采用 Conventional Commits 风格，`<type>(<scope>): <subject>`：

| type     | 说明                                |
| -------- | ----------------------------------- |
| feat     | 新功能                              |
| fix      | Bug 修复                            |
| refactor | 重构（不改外部行为）                |
| docs     | 仅文档                              |
| style    | 格式（不影响代码运行）              |
| test     | 仅测试                              |
| chore    | 构建、依赖、脚本等非功能性变更      |
| perf     | 性能优化                            |

- `scope` 使用后端 app 名（`ops`、`aiops`、`rbac`...）或前端目录（`frontend`）。
- `subject` 中文或英文均可，但保持简洁，不带句号。
- 一个提交只做一件事；超过 200 行改动的拆分提交。

### 3.2 提交粒度

- 一次 PR 只解决一个 Issue 或一个明确目标。
- 数据库迁移单独一个提交；如果同时改了模型、改了视图、改了数据迁移，拆成三个提交。
- UI 改动与对应的接口改动拆开提交；同名提交加 `(api)` / `(web)` 后缀。

## 4. 变更日志

每个 PR 必须同步更新 `CHANGELOG.md`（如果不存在则新建），格式：

```markdown
## [未发布] - YYYY-MM-DD

### 新增
- <功能简述>（对应 PR 链接或提交哈希）

### 修复
- <问题简述>

### 变更
- <接口/字段/行为变化>

### 移除
- <已下线能力>

### 安全
- <鉴权/凭据/限流变更>
```

维护者合并 `test → main` 后，将 `[未发布]` 改写为正式版本号（如 `v0.2.0`）。

## 5. 二次开发切入点建议

按开发难度由低到高排列；详细位置可参考仓库 `README.md` 与 `AGENTS.md`。

| 难度 | 切入点 | 说明 |
| --- | --- | --- |
| 低 | 权限码与内置角色 | `backend/rbac/registry.py`，运行期自动同步 |
| 低 | 数据源类型 | `backend/ops/observability_views.py`，UI 加 Tab |
| 低 | 中间件模板 | `backend/marketplace/template_presets.py` + `seed_templates` |
| 低 | 告警通知渠道 | `backend/ops/alerting.py` |
| 中 | 新智能体 Action | `backend/aiops/action_handlers.py` + `BUILTIN_ACTION_REGISTRY` |
| 中 | 新审批工单类型 | `backend/ops/views.py` + `frontend/src/utils/workorderFlows.js` |
| 中 | 新云平台 SDK | `backend/multicloud/sdk_adapters.py` |
| 高 | 双阶段调度核心 | `backend/aiops/services.py` |
| 高 | CMDB 模型与同步器 | `backend/cmdb/{models,sync}.py` |
| 高 | WebSocket 协议 | `backend/ops/{ssh,k8s_exec}_consumer.py` |

## 6. 关键约束

1. **后端配置默认走环境变量**：`backend/config.json` 仅作为本地开发兜底，生产部署必须使用 `.env` 或编排注入。
2. **后端 RBAC 先于前端呈现**：所有涉及权限、菜单、按钮、WebSocket 的变更，先在 `backend/rbac/registry.py` 注册权限码，再同步前端展示。前端隐藏只是镜像，不是安全边界。
3. **`apply-patches.sh` 必须在 `docker compose up` 之前跑**：品牌资源不在 git 内，`patches/` 目录缺失会让容器缺失前端资产。
4. **飞书风格 UI 规约**：管理台与控制台遵循 `hero + stats cards + 紧凑提示条 + tabs/content` 节奏，详情见 `AGENTS.md`。
5. **不开源机密**：生产密钥、真实 Token、Kubeconfig、SSH 私钥、客户数据、主机凭据不得进入版本库，详见 `OPEN_SOURCE_CHECKLIST.md`。
6. **不走 `pip/npm/apt install`**（参见内网环境约束）：本仓库已声明 `requirements.txt` 与 `package.json`，日常开发在已有的虚拟环境与 `node_modules` 内进行；新增依赖请先在 README 或 `.env.example` 中说明手动安装路径，再以 PR 形式提交。

## 7. 回滚与热修

紧急修复可走 hotfix 流程：

```bash
git switch main
git switch -c hotfix/<topic>
# 最小化修复 + 自测 + 写 changelog
git commit -m "fix(<scope>): <hot-fix subject>"
git push -u origin hotfix/<topic>
# 1) 直接 cherry-pick / 合并回 main
# 2) 然后同步回 test，避免后续升级引入同样的故障
git switch test
git cherry-pick <hotfix-commit>
git push origin test
```

`hotfix/*` 不经过 test 的常规回归周期，但需要维护者与另一位审查者双签才能合并到 `main`。
