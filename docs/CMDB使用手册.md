# CMDB 使用手册

## 1. 模块概览

当前版本的 CMDB 已从“配置项管理页”升级为统一资产治理入口，核心模块包括：

- `配置项管理`：维护 CI 类型、配置项台账、统一资源树和基础属性
- `主机管理`：承载原独立“主机管理”能力，拆分为 `主机资产` 与 `主机申请` 两个子页签
- `资源地图`：查看 CI 关系、拓扑结构和资源依赖
- `成本分析`：从业务线、环境、类型、供应商等维度汇总成本
- `资源优化`：输出资源回收、缩容、定时启停等治理建议

当前产品形态强调两点：

1. 主机治理正式并入 CMDB，不再作为左侧菜单里的独立模块维护
2. 主机管理与配置项管理共用同一套业务资源树，业务线与环境口径保持一致

---

## 2. 配置项管理

### 2.1 统一资源树

配置项管理左侧采用“业务线 -> 环境”的统一资源树：

- 点击业务线节点，可查看该业务线下所有配置项
- 点击环境节点，可进一步筛选到具体环境
- 点击树标题，可清空树筛选，恢复全量视图

这套树结构同时复用于 `主机资产` 与 `主机申请`，因此三块数据在业务线和环境维度上天然对齐。

### 2.2 配置项台账

配置项列表支持按以下维度筛选和查看：

- CI 类型
- 业务线
- 环境
- 状态
- 名称 / IP / 负责人搜索

典型字段包括：

- 配置项名称
- CI 类型
- 业务线 / 环境
- IP 地址
- 规格信息
- 月成本
- 负责人
- 状态

适合沉淀应用、数据库、中间件、容器平台、网络入口等基础台账。

## 3. 主机管理

`主机管理` 是本次 CMDB 调整的重点，作为主 Tab 放在 CMDB 内部，并拆分为两个子页签：

- `主机资产`
- `主机申请`

### 3.1 主机资产

主机资产页用于统一纳管已交付或已接入平台的主机。

#### 页面能力

- 左侧使用与配置项管理一致的 `主机资源树`
- 支持按主机名 / IP / 负责人搜索
- 支持按状态筛选：`在线 / 离线 / 告警`
- 支持查看 CPU、内存、磁盘使用率
- 支持查看 SSH 用户和端口
- 支持 `测试连接`、`刷新信息`、`终端`、`编辑`、`删除`

#### 典型台账字段

- 主机名
- IP 地址
- 业务线
- 环境
- 负责人
- 操作系统
- 在线状态
- CPU / 内存 / 磁盘使用率
- SSH 连接信息
- 描述

#### Demo 数据设计原则

当前 Demo 主机资产已按典型场景重做，重点保证和配置项管理对得上：

- 与配置项管理共用相同的业务线和环境节点
- 保留少量但典型的生产 / 测试 / 开发主机场景，便于演示
- 主机命名、IP、负责人、用途描述与 CMDB 中的服务和组件关系保持一致

这意味着在演示时可以先看配置项，再切到主机资产查看同一业务域下的承载主机。

### 3.2 主机申请

主机申请页是资源申请能力的收口版本，当前只保留“申请主机”这一种申请类型。

#### 页面能力

- 支持新建主机申请
- 支持按状态筛选：`待审批 / 已批准 / 已拒绝 / 已完成`
- 支持按标题 / 主机名 / IP / 申请人搜索
- 支持审批、拒绝、交付完成
- 已批准申请可一键 `转主机资产`

#### 申请字段

- 申请标题
- 主机规格说明
- 业务线 / 环境
- 主机名 / IP
- 操作系统
- 负责人
- 优先级
- 申请原因

#### 审批与交付流转

当前主机申请的标准流程如下：

1. 申请人提交主机申请
2. 审批人执行批准或拒绝
3. 已批准申请在交付完成后执行“转主机资产”
4. 系统将申请信息写入主机资产台账，形成闭环

这样演示时可以清楚体现：

- 申请前是需求
- 批准后是待交付
- 完成后进入正式资产

### 3.3 主机资产与主机申请的联动

主机管理这次调整不只是页面合并，还补上了资产与申请之间的数据联动：

- 主机申请的业务线 / 环境来源于统一资源树
- 主机申请完成后可直接转为主机资产
- 主机资产与主机申请的 Demo 数据相互对应，便于演示“申请 -> 审批 -> 交付 -> 纳管”

## 4. 资源地图

资源地图用于查看和维护配置项之间的关系，包括：

- `runs_on`：运行在
- `connects_to`：连接到
- `depends_on`：依赖

常见用法：

- 从某个应用服务出发查看其依赖的数据库或中间件
- 从某台主机出发查看其承载的服务或组件
- 从某个核心中间件出发评估故障影响面

资源地图适合和主机资产、配置项管理配合使用，完成“台账 -> 关系 -> 拓扑”的链路分析。

---

## 5. 成本分析与资源优化

### 5.1 成本分析

成本分析页会按以下维度输出汇总结果：

- 月度总成本
- 业务线成本分布
- 环境成本分布
- 资源类型成本分布
- 供应商成本分布
- Top 成本资源

### 5.2 资源优化

资源优化页基于成本、规格、使用率和状态给出治理建议，例如：

- 可回收资源
- 非生产环境定时启停建议
- 高成本低利用率缩容建议
- 存储优化建议
- 归属补齐建议

## 6. 推荐演示顺序

如果你要向团队、客户或面试官演示 CMDB，推荐按下面顺序展开：

1. 先打开 `配置项管理`，说明统一资源树和 CI 台账
2. 切到 `主机管理 / 主机资产`，展示主机纳管、监控指标、终端入口
3. 切到 `主机管理 / 主机申请`，展示申请、审批和转资产流程
4. 打开 `资源地图`，说明配置项与主机、服务、中间件之间的关系
5. 最后查看 `成本分析` 和 `资源优化`，说明治理闭环

---

## 7. 当前版本最适合强调的价值

当前这版 CMDB 最适合强调的不是单一台账，而是统一治理闭环：

- 配置项沉淀基础资源
- 主机资产承接实际主机台账
- 主机申请承接资源交付流程
- 资源地图负责关系治理
- 成本分析与资源优化负责持续治理

如果需要对外汇报，本次版本最值得强调的一句话是：

> AgDevOps 已把主机纳管、主机申请和配置项治理统一收口到 CMDB，形成从申请、交付到资产沉淀的完整闭环。
