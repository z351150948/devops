# 链路追踪接入与展示实现思路

## 目标

链路追踪模块定位为 SxDevOps 的统一排障入口，而不是重新实现一套完整的 Trace 存储引擎。平台负责统一接入、查询、展示和权限管控，底层 Trace 数据仍由 SkyWalking、OpenTelemetry Collector、Tempo、Jaeger、Zipkin 等组件采集与存储。

核心目标：

- 同时支持 SkyWalking 和 OpenTelemetry 生态数据。
- 前端体验对齐 Jaeger、Zipkin、Tempo 的常见排障视图。
- 后端使用统一 Trace 模型屏蔽不同数据源差异。
- 支持后续和日志、告警、CMDB、K8s、发布系统联动。

## 接入架构

推荐采用“统一查询层 + Provider 适配器”的方式：

```text
应用 / 中间件 / 网关
        │
        ├── SkyWalking Agent ──> SkyWalking OAP ──> SxDevOps SkyWalking Provider
        │
        └── OpenTelemetry SDK ─> OTel Collector ─┬─> Tempo  ─┐
                                                  ├─> Jaeger ├─> SxDevOps OTel Provider
                                                  └─> Zipkin ┘
```

OpenTelemetry 数据不建议直接写入 Django。推荐通过 OpenTelemetry Collector 做采集、清洗、采样和转发，再落到 Tempo、Jaeger 或 Zipkin 这类专业后端，SxDevOps 通过查询接口消费数据。

## Provider 规划

第一阶段内置以下 Provider：

- `demo`：无外部配置时使用演示数据，便于本地开发和页面验收。
- `skywalking`：通过 SkyWalking OAP GraphQL 查询服务、Trace、Span 和拓扑。
- `tempo`：面向 OpenTelemetry Collector + Grafana Tempo，支持 TraceQL/Tempo 查询接口和 OTLP Trace 结构归一化。
- `jaeger`：面向 OpenTelemetry Collector + Jaeger Collector/Query，支持 Jaeger Query HTTP API。
- `zipkin`：面向 OpenTelemetry Collector + Zipkin，支持 Zipkin v2 HTTP API。

后续可扩展为数据库配置型 Provider，让用户在页面上动态维护数据源；当前先使用环境变量和 `OBSERVABILITY_CONFIG` 做静态配置。

## 统一数据模型

前端只消费统一结构，避免页面绑定某个厂商字段。

### Trace

- `trace_id`：链路 ID。
- `service_id` / `service_name`：入口服务。
- `endpoint_names`：入口接口或关键操作。
- `duration_ms`：总耗时。
- `start`：开始时间。
- `is_error` / `state`：错误状态。
- `span_count`：Span 数量。
- `source_provider`：来源 Provider。

### Span

- `span_id`：Span ID。
- `parent_span_id`：父 Span ID。
- `service_code`：服务名。
- `service_instance_name`：实例名。
- `endpoint_name`：操作名。
- `start_time` / `end_time`：起止时间。
- `duration_ms`：耗时。
- `type`：Span 类型或 Kind。
- `component`：组件，例如 HTTP、MySQL、Redis。
- `peer`：远端地址。
- `is_error`：是否异常。
- `layer`：HTTP、RPC、DATABASE、CACHE 等层级。
- `tags`：SkyWalking tags 或 OTel attributes。
- `logs`：SkyWalking logs 或 OTel events。

## 页面规划

链路追踪页按现有管理页规范落地：

- 顶部 `hero` 只保留主标题和操作按钮。
- 使用 `release-stat-card` 展示服务数、Trace 命中数、错误链路、拓扑调用。
- 保留紧凑运行提示条，说明当前 Provider、演示/实时状态和回退信息。
- 使用标签页组织功能：`Trace 检索`、`服务拓扑`、`外部入口`。

Trace 检索页重点对齐 Jaeger/Zipkin/Tempo：

- 支持 Provider、服务、状态、时间窗口、关键字、Trace ID、数量过滤。
- Trace 列表展示入口服务、入口接口、耗时、状态、开始时间。
- Trace 详情展示概要指标、异常 Span、慢 Span、服务分布。
- 增加瀑布图视图，支持按父子层级缩进、耗时比例条、错误/慢调用高亮。
- Span 卡片展示 tags、logs/events，便于直接定位错误原因。

## 权限与审计

链路追踪属于只读排障能力，继续复用现有权限：

- 页面入口：`ops.trace.view`
- 后端查询：`ops.trace.view`
- 日志联动：`ops.log.query`
- 告警入口：`ops.alert.view`
- Grafana 入口：`ops.grafana.view`

当前阶段不新增写操作，因此不接入事件墙审计。后续如果增加“保存查询模板、修改数据源、触发采样策略”等写操作，应新增 `ops.trace.manage` 并记录事件墙。

## 配置规划

建议环境变量：

- SkyWalking：`SKYWALKING_ENABLED`、`SKYWALKING_UI_URL`、`SKYWALKING_OAP_URL`、`SKYWALKING_GRAPHQL_PATH`
- Tempo：`TEMPO_ENABLED`、`TEMPO_QUERY_URL`、`TEMPO_UI_URL`
- Jaeger：`JAEGER_ENABLED`、`JAEGER_QUERY_URL`、`JAEGER_UI_URL`
- Zipkin：`ZIPKIN_ENABLED`、`ZIPKIN_QUERY_URL`、`ZIPKIN_UI_URL`
- 默认 Provider：`TRACING_DEFAULT_PROVIDER`

## 实施节奏

1. 先抽象后端 Provider 查询层，保持原 API 路径兼容。
2. 加入 Tempo、Jaeger、Zipkin 的基础服务查询、Trace 查询和详情归一化。
3. 改造前端链路追踪页，增加 Provider 选择、功能标签页和瀑布图。
4. 保留演示数据兜底，外部系统未配置或不可达时仍可验收页面。
5. 后续补数据源管理、连接测试、Trace 与日志/告警/资源的深度关联。
