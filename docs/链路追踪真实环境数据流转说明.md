# 链路追踪真实环境数据流转说明

## 1. 目标

本文记录 SxDevOps 在接入真实链路平台后，Trace 数据是如何查询、归一化并最终绘制到页面上的，便于后续交接、排障和继续扩展。

SxDevOps 自身不负责采集和存储 Trace，而是作为统一查询与展示层，复用已有的链路后端能力。

## 2. 总体链路

```text
业务应用
  ↓
埋点 / 自动探针
  ├─ SkyWalking Agent
  └─ OpenTelemetry SDK / Java Agent / Collector
  ↓
链路采集与存储后端
  ├─ SkyWalking OAP
  ├─ Jaeger
  ├─ Zipkin
  └─ Tempo
  ↓
各平台查询接口
  ├─ SkyWalking GraphQL API
  ├─ Jaeger Query API
  ├─ Zipkin v2 API
  └─ Tempo Query API
  ↓
SxDevOps 后端 Provider 适配层
  ↓
统一 Trace / Span / Service / Topology 数据模型
  ↓
SxDevOps 前端链路追踪页面
  ├─ 左侧 Trace 列表
  ├─ 顶部 Trace 概览
  ├─ Trace Timeline
  ├─ Span Waterfall
  ├─ Span 详情
  └─ 服务调用拓扑
```

## 3. 采集与存储边界

- 采集不由 SxDevOps 实现，交给业务应用内的探针或 SDK。
- 存储不由 SxDevOps 实现，交给 SkyWalking、Jaeger、Zipkin、Tempo 等链路后端。
- SxDevOps 主要做三件事：
  - 维护数据源配置
  - 调用各平台查询接口
  - 统一渲染成一套 Trace UI

## 4. 真实环境接入方式

### 4.1 SkyWalking

- 应用侧通常接入 `SkyWalking Agent`
- 数据先进入 `SkyWalking OAP`
- SxDevOps 通过 `SkyWalking OAP GraphQL` 查询：
  - 服务列表
  - Trace 列表
  - Span 详情
  - 全局拓扑

代码落点：

- `backend/ops/tracing_providers.py` 中：
  - `_load_skywalking_services`
  - `_load_skywalking_topology`
  - `_search_skywalking_traces`
  - `_load_skywalking_trace_detail`

### 4.2 Jaeger

- 应用可通过 `OpenTelemetry` 或 `Jaeger` 协议上报
- 数据进入 `Jaeger`
- SxDevOps 通过 `Jaeger Query API` 查询：
  - 服务列表
  - Trace 列表
  - Trace 详情

代码落点：

- `backend/ops/tracing_providers.py` 中：
  - `_load_jaeger_services`
  - `_search_jaeger_traces`
  - `_load_jaeger_trace_detail`

### 4.3 Zipkin

- 应用可通过 `OpenTelemetry` 或 Zipkin 兼容方式上报
- 数据进入 `Zipkin`
- SxDevOps 通过 `Zipkin v2 API` 查询：
  - 服务列表
  - Trace 列表
  - Trace 详情

代码落点：

- `backend/ops/tracing_providers.py` 中：
  - `_load_zipkin_services`
  - `_search_zipkin_traces`
  - `_load_zipkin_trace_detail`

### 4.4 Tempo / OpenTelemetry

- 应用通常通过 `OpenTelemetry SDK / Agent`
- 数据先进入 `OpenTelemetry Collector`
- 再由 Collector 转发到 `Tempo`
- SxDevOps 通过 `Tempo Query API` 查询：
  - 服务标签
  - Trace 列表
  - Trace 详情

代码落点：

- `backend/ops/tracing_providers.py` 中：
  - `_load_tempo_services`
  - `_search_tempo_traces`
  - `_load_tempo_trace_detail`

## 5. 为什么不同平台都能画成同一种 UI

不同链路平台的返回格式并不一致：

- SkyWalking 偏 GraphQL 风格
- Jaeger / Zipkin 偏 REST JSON
- Tempo 偏 OpenTelemetry / OTLP 结构

SxDevOps 后端会先做一层统一归一化，把不同平台结果转换成统一模型，再交给前端渲染。

统一转换核心包括：

- Trace 基础信息
  - `trace_id`
  - `service_name`
  - `endpoint_names`
  - `duration_ms`
  - `start`
  - `is_error`
- Span 基础信息
  - `span_id`
  - `parent_span_id`
  - `service_code`
  - `endpoint_name`
  - `start_time`
  - `end_time`
  - `duration_ms`
  - `type`
  - `layer`
  - `tags`
  - `logs`

代码落点：

- 统一 Trace 详情整理：`backend/ops/tracing_providers.py` 中 `_trace_detail_from_spans`
- 统一拓扑拼装：`backend/ops/tracing_providers.py` 中 `_build_topology_from_trace_details`

## 6. 页面是怎么画出来的

### 6.1 前端不直接连外部链路平台

前端只调用 SxDevOps 自己的后端接口，不直接请求 SkyWalking、Jaeger、Zipkin、Tempo。

这样做的好处：

- 避免前端暴露外部链路平台地址和认证信息
- 屏蔽不同平台接口差异
- 统一权限控制与错误处理

### 6.2 页面数据来源

前端页面：

- `frontend/src/views/TraceObservability.vue`

前端主要调用：

- `getTracingCatalog`
- `searchTracing`
- `getTraceDetail`

对应后端入口：

- `backend/ops/observability_views.py`
  - `observability_tracing_catalog`
  - `observability_tracing_search`
  - Trace 详情接口

### 6.3 渲染过程

后端返回统一结构后，前端将其拆成不同视图：

- 左侧 Trace 列表：按时间或筛选条件展示命中链路
- 顶部概览：展示 Trace ID、核心指标、Timeline 摘要
- Waterfall：按 Span 父子层级和起止时间画瀑布图
- Span 详情：展示 tags、events、logs
- 服务拓扑：展示服务间调用关系

## 7. 一句话总结

SxDevOps 的链路追踪页面，本质上是：

**对 SkyWalking、Jaeger、Zipkin、Tempo 等真实链路平台的查询结果做统一适配，再用一套统一 UI 进行展示。**

不是自己做 Trace 采集器，也不是自己做 Trace 存储引擎。
